Device Registration & Inventory

  • Quick facts at registration: friendly computer name, hardware model, OS version, console user, FQDN — the server derives the IP itself (client-claimed addresses would be spoofable).
  • Process inventory via libproc: proc_listallpids + proc_pidinfo (start time, uid) + proc_pidpath + KERN_PROCARGS2 for the full command line; GUI apps sorted first, capped at 300.
  • Browser-extension inventory: Chrome/Edge/Brave manifest scan (with __MSG_ locale-name resolution) + Firefox extensions.json.
  • Cadence: registered + snapshot at startup, refreshed every 10 minutes when VG_UPLOAD_URL is set; vguardiand --inventory-once for a one-shot run (installers/verification). Never blocks enforcement.

Security Posture

  • Privacy: Audit events built from engine's redacted response; raw prompts never queued or uploaded.
  • Fail-closed: Unavailable engine → no decision; missing policy → WARN; unsigned policy rejected; XPC peers rejected without code requirement.
  • At rest: Event signing key + identity at 0600 permissions; SQLCipher (AES-256) for queue DB.

Detection Categories (24)

Deterministic, redaction-safe detectors cover every category below; wire names are snake_case and synchronized across the engine, gRPC proto, Swift agent, connector SDK, backend, and dashboard.

Decision Pipeline

detectors (24 categories) ─► risk score 0–100 + tier (safe / low / sensitive / confidential / restricted) │ ├─► Granite Guardian 3.0 2B (optional, llama.cpp) — raise-only tier │ refinement + category attribution; grammar-constrained output ├─► Code classifier (optional, fine-tuned CodeBERT) — only when the │ source-code/config gate fired; sensitive ≥0.8 → Confidential floor ▼ signed-bundle rule evaluation (DSL, ed25519) ─► critical force-block ▼ ALLOW / WARN / BLOCK + redacted findings + reason
  • Raise-only & fail-open refinement: the LLM and code classifier can only raise the rule-based risk tier, never lower it; any transport/parse failure falls back to the deterministic score.
  • Critical force-block: any Critical finding (AWS key, destructive command, credentialed connection string) overrides an Allow/Warn rule outcome — unless an approved, time-bounded exception applies.
  • Config: VG_DETECTOR_CONFIG (YAML, unknown keys rejected) for org keyword lists, domain phrase lists, thresholds; VG_LLM_ENDPOINT/VG_LLM_TIMEOUT_MS (default 2500 ms) and VG_CODE_CLASSIFIER_ENDPOINT/VG_CODE_CLASSIFIER_TIMEOUT_MS for the optional refinement stages.

🧠 Granite Guardian 3.0 2B (self-hosted)

  • IBM's purpose-built risk/safety classifier (Apache-2.0)
  • Served locally via llama.cpp (GGUF Q4_K_M, ~1.6 GB) — data never leaves your infrastructure
  • ./provision-policy-rules.sh downloads the model; compose profile llm serves it on :8090
  • Verdict cache (prompt-hash) amortizes repeat scans

⚡ Policy Actions

  • ALLOW Prompt forwarded normally
  • WARN Low risk: notice · Medium: interactive acknowledge · High/Critical: hard block
  • BLOCK Prompt prevented from leaving the device (fail-closed when any hop is down)

Audit Integrity — Tamper-Evident Chain

Each event has event_hash = SHA-256(canonical_content ‖ previous_event_hash), linked per device. Chain advance + deduplication is a single DynamoDB TransactWriteItems (atomic). upload_id replays provide whole-batch idempotency; event_id conditional writes prevent duplicates.

14. Process Flow — Step by Step

This section traces exactly how a request flows through VGuardrail, step by step, in the two ways the system runs: the Local stack (single machine, no AWS — for development and demos) and the Production / real-time deployment (endpoint agents + AWS cloud). The components are the same; what differs is the enforcement path (HTTP /scan vs. XPC→gRPC), the authentication (dev headers vs. Cognito JWT), and where audit data lands (in-memory store vs. DynamoDB + S3 hash chain).

A. Local Running — flow (single machine, no AWS)

Goal: run the entire detection + dashboard pipeline on one Mac with no AWS account. Authentication is bypassed (VG_DEV_CLAIMS=1 / DISABLE_AUTH=true); detection runs in-process inside the backend via the POST /scan endpoint.

A.1 — Bring-up sequence (what starts, in order)

  1. ./start-local.sh launches the local Docker stack from docker-compose.local.yml: dynamodb-local:8000 minio:9000 mock-jwks backend:8080 (VG_DEV_CLAIMS=1). The backend builds the in-process DetectorRegistry (24 categories) once at startup.
  2. Dashboard starts on localhost:3000 via npm run dev with DISABLE_AUTH=true (mock super_admin; no Cognito).
  3. Endpoint chain (optional, for agent/CLI tests) — two user LaunchAgents: pe-engined on the UDS /tmp/vguardrail/policy.sock and vguardiand owning the XPC service com.vguardrail.agent.xpc.
  4. (Optional) Granite Guardian LLM — ./provision-policy-rules.sh then the llm compose profile; set VG_LLM_ENDPOINT to enable raise-only refinement.
  5. ./seed-local-data.sh loads demo devices / policies / users / inventory so the dashboard has content.

A.2 — Runtime request flow (one prompt, end to end)

  1. User submits a prompt in a connected surface — a browser AI site (ChatGPT/Claude/ Gemini), an IDE command, or a vg-* CLI wrapper.
  2. Connector intercepts pre-send. The browser content-script gate cancels the submit at the capture phase; the CLI wrapper extracts the prompt before exec; the IDE/Claude hook fires on submit. Nothing has left the machine yet.
  3. Connector calls POST /scan on the backend http://localhost:8080/scan with headers x-vg-org-id + x-device-id (JWT skipped because VG_DEV_CLAIMS=1). Any attached file is sent base64 in files[].
  4. Backend runs the real detector pipeline in-process: DetectorRegistry.scan_all → classify_risk → optional Granite Guardian refine (raise-only). File attachments are extracted first (PDF / Office / ZIP) under an 800 ms hybrid cap and folded into the same evaluation.
  5. A Decision is returned — action, risk_score, confidence, findings, reason.
  6. The connector enforces locally: ALLOW forward the prompt · WARN low = notice, medium = "Send anyway", high = hard block · BLOCK the prompt never leaves the device.
  7. The backend records an audit event in its in-memory dev store (DEV_AUDIT_STORE) — redacted findings only, never raw content.
  8. The dashboard reads it via GET /admin/audit / /admin/devices at localhost:3000 — violations, devices, audit.

B. Production / Real-Time Deployment — flow (endpoints + AWS cloud)

Goal: enforce policy on the device in real time (the prompt is evaluated by the local policy engine over XPC→gRPC, so a block never touches the network), and ship a tamper-evident audit trail to the cloud over authenticated HTTPS.

B.1 — Bring-up sequence (one-time / per release)

  1. Provision AWS infra with Terraform — DynamoDB (core + audit), S3 (audit archive, Object Lock), Cognito (user pool + app clients), KMS, Secrets Manager, Route 53 + ACM.
  2. Build & push images to ECR, then deploy backend and dashboard on ECS Fargate behind Caddy (auto Let's Encrypt TLS, rate-limit on api.*).
  3. Install endpoints. On each managed Mac, vguardiand and pe-engined run as signed LaunchDaemons; the signed policy bundle (ed25519) is pinned; browser/IDE/CLI connectors are installed.
  4. Enroll the device. POST /devices/register with an enrollment token → the backend returns a Cognito JWT the agent stores for authenticated uploads.

B.2 — Real-time enforcement flow (one prompt, on the device)

  1. User submits a prompt in a connected app; the connector intercepts it pre-send.
  2. Connector → XPC to vguardiand on com.vguardrail.agent.xpc. The daemon validates the caller's code signature (signed-peer check) before accepting.
  3. vguardiand → gRPC over UDS to pe-engined (/var/run/vguardrail/policy.sock), the local policy engine running the pinned, signature-verified policy bundle.
  4. pe-engined decides in <50 ms: 24-category detectors → risk score → optional LLM refine → Decision. Fail-closed: if any hop (XPC, gRPC, engine) is down, the connector blocks.
  5. Connector enforces on the device: ALLOW · WARN (tiered) · BLOCK — on block the prompt never leaves the machine.
  6. vguardiand signs an audit event (ed25519) and enqueues it in an offline-first SQLite queue — capture and enforcement never depend on the network.
  7. Upload worker batches events and POSTs them over HTTPS to api.verisync.digital/events/batch with the device's Authorization: Bearer <JWT>.
  8. Caddy terminates TLS and proxies to the backend :8080, which verifies the Cognito RS256 JWT against the pool's JWKS before accepting anything.
  9. Backend appends to the tamper-evident hash chain — each event links to the prior (hi = SHA-256(ei ‖ hi-1)) — writing the chain head + records to DynamoDB and the raw signed batch to S3 (Object Lock).
  10. Security team observes via the dashboard: Caddy → Next.js BFF → Cognito login → backend /admin/* (RBAC) → devices, violations, exceptions, and audit-chain verification.

Production splits responsibilities: enforcement is real-time and local (XPC→gRPC, decision in <50 ms, fail-closed) while audit is authenticated and durable (offline-first queue → JWT-verified HTTPS → hash-chained DynamoDB + S3).

18. Coding Standards & Best Practices
